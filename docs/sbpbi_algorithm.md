# ApproximateSbpbi アルゴリズム解説

Labarre (2018) の Algorithm 1「ApproximateSbpbi」の実装解説。

---

## 1. 問題設定

**Sorting by Prefix Block Interchanges (SBPBI)**

- 入力: 1-indexed の順列 `arr = [a_1, a_2, ..., a_n]`（値は 1〜n の置換）
- 操作: **プレフィックスブロック交換** BI(i, j, k, l)
  - 区間 `[i, j)`, `[j, k)`, `[k, l)` を `[k, l)`, `[j, k)`, `[i, j)` に並び替える
  - ただし必ず `i = 1`（プレフィックスから始まる）

```
1=i        j        k        l              n
 |←block1→|←block2→|←block3→|←(unchanged)→|

         ↓ BI(i, j, k, l)

1=i        j        k        l              n
 |←block3→|←block2→|←block1→|←(unchanged)→|
```
- 目標: 最小手数で恒等順列 `[1, 2, ..., n]` にする
- この実装は **2-近似アルゴリズム**（最適解の最大2倍以内の手数で必ず終了）

---

## 2. 主要データ構造

### ブレークポイントグラフ (`t_bp_graph`)

サイズ `2*(n+1)` の配列で、各要素が頂点1つに対応する。

```
要素 k (1 ≤ k ≤ n) に対して:
  left(k)   = 頂点 2k-1   （要素の左端）
  right(k)  = 頂点 2k     （要素の右端）
  left(0)   = 頂点 0      （左番兵）
  right(n+1)= 頂点 2n+1   （右番兵）
```

各頂点は3つのフィールドを持つ：

| フィールド | 意味 |
|---|---|
| `grey`     | 恒等順列における隣接頂点（固定・不変）|
| `black`    | 現在の順列における隣接頂点（操作のたびに更新）|
| `cycle_id` | 属するサイクルの番号 |

**グレー辺**（`set_greys`）: 偶数頂点 v → v+1、奇数頂点 v → v-1  
→ ソート済み状態での隣接関係を表す（固定）

**ブラック辺**（`set_blacks`）: 現在の順列に基づく隣接関係  
→ `right(arr[i])` と `left(arr[i+1])` を結ぶ（操作のたびに部分更新）

### サイクル分解

グレー辺とブラック辺を交互にたどることでできる閉路の集合。  
サイクル数 = `n+1` のとき、順列が恒等順列（ソート完了）。  
BI(i,j,k,l) を適用するとサイクル数は **1または2増加**する。

### tmp 配列

`ctx->tmp` は `2*(n+1)` サイズの整数配列で、**2つの用途を兼ねる**：

1. **頂点インデックス変換テーブル**（`refresh_tmp`）  
   位置 p に値 v がある → `tmp[2p-1] = 2v-1`, `tmp[2p] = 2v`  
   これにより「位置 p の left/right 頂点番号」が O(1) で引ける

2. **ブロック交換時のバッファ**（`block_interchange` 末尾の memcpy）  
   配列の一部を一時退避する作業領域として再利用

---

## 3. アルゴリズムの流れ

```
approx_sbpbi(ctx)
│
├─ set_cycle_id() でサイクル数 o_count を計算
│
└─ while o_count < n+1:          ← ソート完了までループ
    │
    ├─ refresh_tmp()              ← tmp に現在の順列の頂点番号を記録
    │
    ├─ if arr[0] != 1:            ← 先頭が1でない（パターンA/B）
    │   ├─ find_pattern_a() で BI を探す
    │   │   成功 → サイクル数 +2 が期待できる操作
    │   └─ 失敗 → find_pattern_b() で BI を探す
    │               サイクル数 +1 が保証される操作
    │
    └─ if arr[0] == 1:            ← 先頭が1（パターンC）
        └─ find_pattern_c() で BI を探す
            先頭ブロックを延ばす操作
    │
    └─ block_interchange() で実際に操作を適用
       ├─ ブラック辺を4本更新（グラフ差分更新）
       └─ memcpy 3回で配列の区間を並び替え
    │
    └─ set_cycle_id() でサイクル数を再計算
```

---

## 4. 3つのパターン詳細

### パターン A（`find_pattern_a`）

**適用条件**: `arr[0] != 1` かつ適切な頂点ペアが存在する

1. `first = tmp[1]`（現在の先頭要素の left 頂点）
2. `first - 1` の位置 j を tmp 内で探す（恒等順列での「前の要素の right」）
3. `tmp[i] + 1 == tmp[k]` かつ `cycle_id == 0` を満たす i, k を探索
4. 見つかれば BI(1, i/2+1, j/2+1, k/2+1) を設定

`cycle_id == 0` のチェックにより、サイクルを跨ぐ操作（+2 増加）を優先的に選ぶ。

### パターン B（`find_pattern_b`）

**適用条件**: パターン A が失敗

1. パターン A と同様に j を求める
2. `tmp[i] - 1 == tmp[k]` を満たす i, k を探索（条件が緩い）
3. 必ず BI が見つかり、サイクル数が少なくとも +1 される

### パターン C（`find_pattern_c`）

**適用条件**: `arr[0] == 1`（先頭要素は正位置）

1. 先頭から連続している区間の末尾 i を見つける（`arr[i+1] != arr[i]+1`）
2. その次に来るべき値 `target = arr[i]+1` の位置 j を探す
3. BI(1, i+2, i+2, j+1) で target を先頭ブロックに引き込む

---

## 5. ブロック交換の実装（`block_interchange`）

BI(i, j, k, l) の適用で変わるブラック辺は **4本だけ**。

`j != k`（3ブロックが全て非空）の場合：

```
A: right(i-1) <-> left(k)    （先頭ブロックの左境界 → ブロック3の左端）
B: right(l-1) <-> left(j)    （末尾ブロックの右境界 → ブロック2の左端）
C: right(k-1) <-> left(i)    （ブロック3の右境界   → ブロック1の先頭）
D: right(j-1) <-> left(l)    （ブロック2の右境界   → 末尾ブロックの右端）
```

`j == k`（中間ブロックが空、実質トランスポジション）の場合：  
辺の接続先が変わり、専用の処理で正しく更新する。

配列の並び替えは tmp をバッファに 3回の `memcpy` で完結：

```c
tmp         ← arr[k-1 .. l-1]   （ブロック3を退避）
tmp+(l-k)   ← arr[j-1 .. k-1]   （ブロック2を退避）
tmp+(l-j)   ← arr[i-1 .. j-1]   （ブロック1を退避）
arr[i-1 .. l-1] ← tmp           （3→2→1 の順に書き戻す）
```

---

## 6. 実装上の工夫

### (1) 頂点番号の直接計算

要素の値 v に対して left = `2v-1`、right = `2v` と定義することで、  
「値 v の頂点番号」が **乗算1回** で求まる。ハッシュマップ等は不要。

### (2) グラフの差分更新

BI のたびにグラフ全体を再構築するのではなく、変化する **4本のブラック辺だけ** を更新する。  
これにより1操作あたりのグラフ更新コストが O(1) になる。

### (3) サイクル数を終了条件に使用

サイクル数が `n+1` に達したとき恒等順列であることが保証されるため、  
明示的に「全要素が正しい位置か」チェックしなくて済む。
